# Hugh — Product Vision

## What Hugh Is

Hugh is a career development platform built around three pillars: **Interview Prep** (current focus), **Progress Tracker** (future), and **Focused Learning** (future). Interview Prep is a voice-driven mock interview trainer where users select a domain room, are assigned a random AI persona, and work through a structured question loop — the persona asks a question aloud via TTS, the user responds by voice, reviews their transcript, submits, and receives spoken feedback from Claude. The name honours the first person who ever interviewed the founder. Progress Tracker will surface session history, readiness scores, and improvement trends over time. Focused Learning will serve targeted drills and concept refreshers between interview sessions. All three pillars share the same auth layer (Supabase) and are designed to compound: Interview Prep generates the performance data that Powers Tracker; Tracker surfaces the gaps that Focused Learning addresses.

---

## Interview Prep — Current State

As of Phase 5, the full interview loop is built and working end-to-end. Users land on a room selection page and choose one of three domains: Data Engineering, Data Science, or ML Engineering. A session is created in Supabase, a persona is randomly assigned from three options (Marcus, Sarah, James — each with a distinct ElevenLabs voice), and the interview begins immediately with a TTS-played question.

The core state machine (`useInterview.ts`) enforces a strict sequence: `IDLE → PLAYING_QUESTION → READY → RECORDING → REVIEWING → SUBMITTING → FEEDBACK → NEXT | BREAK`. Questions are generated by Claude — a small hardcoded pool of intro questions for the first one or two turns, then dynamically generated domain questions for the remainder. The best answer for each question is also Claude-generated and stored server-side. During READY state, users can request a one-sentence hint or reveal the ideal answer before recording. Speech capture uses the browser-native Web Speech API (Chrome/Edge only). After submission, Claude runs a similarity check against the ideal answer (≥ 90% alignment = "used best answer") and generates 2–3 sentence spoken feedback. Two coaching modes exist: **active** (full feedback loop after each answer) and **passive** (silent processing, feedback stored but not surfaced, jumps straight to the next question). Sessions can be paused and are persisted to Supabase with status tracking.

The UI is a fixed three-zone layout — QuestionZone (capped at 35 vh), a scrollable content zone, and ActionZone anchored at the bottom — with no page-level scroll. Tailwind CSS, dark slate theme, Lora serif for question text.

## Interview Prep — Next Version Priorities

1. **Job ad input → tailored question generation.** Allow the user to paste a job description before starting. The question generation prompt uses it to weight questions toward the specific stack, seniority level, and responsibilities in that posting. No new UI beyond a text area on the session-start screen.

2. **Custom topic selection.** Let users define their own interview domain or sub-topic beyond the three fixed rooms (e.g. "Kafka + Flink pipelines", "causal inference", "LLM fine-tuning"). The room selector becomes a free-text field with the three presets as defaults.

3. **Toggle to skip intro question.** A session-start option for returning users who want to go straight to domain questions. Skipping intro means `questionType` starts as `'domain'` and `questionIndex` starts at 2 (past the hardcoded intro pool threshold).

4. **Speed improvements.** Two parallel tracks: (a) stream TTS audio so playback starts before the full clip is generated; (b) pre-fetch the next question in the background during FEEDBACK state so the PLAYING_QUESTION transition is instant after "Next Question" is clicked.

5. **Performance dashboard.** A post-session summary screen and a persistent history view, both reading from the `answers` and `sessions` tables. Requires Supabase production to be configured and RLS policies confirmed. Blocked until production environment variables are set.

## Architecture Constraints

- **Next.js 15 App Router** — server components for data fetching and session creation; client components only where interactivity requires it. API routes live under `app/api/interview/` and are the only place external services are called.
- **Supabase** — PostgreSQL for persistence, Supabase Auth for login/signup. A dev-bypass (`auth-helper.ts`) is active so API routes work without a live session during local development. Production Supabase is not yet configured; migrations must be run manually in the Supabase Dashboard before launch.
- **Anthropic Claude (`claude-sonnet-4-6`)** — question generation, best-answer generation, similarity check (LLM-judged, not string match), feedback generation, hint generation, and session assessment. All calls are server-side only.
- **ElevenLabs** — TTS for all spoken audio. Cost-sensitive: the `tts` route enforces a 2 000-character hard cap (returns 400 if exceeded). Three voice IDs map to the three personas via server-only environment variables.
- **Web Speech API** — browser-native STT, continuous mode with interim + final transcript merging. Chrome and Edge only; other browsers see a warning but the rest of the app remains functional.
- **Secrets policy** — `ANTHROPIC_API_KEY`, `ELEVENLABS_API_KEY`, all three `ELEVENLABS_VOICE_ID_*` values, and `SUPABASE_SERVICE_ROLE_KEY` never leave the server. `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` are the only values safe for the client.

## What Not To Break

- **`ClientPersona` pattern** — `Persona` has a `voiceId` field; `ClientPersona` is `Omit<Persona, 'voiceId'>`. The server resolves `voiceId` from `personaId` at the `/api/interview/tts` route. `ClientPersona` is what gets passed to client components. Never change this to pass the raw `Persona` to the browser.
- **Three-zone layout** — `QuestionZone` (flex-none, max-h-[35vh], internal scroll) / content zone (flex-1, min-h-0, internal scroll per state) / `ActionZone` (flex-none, h-28, always anchored at bottom). The outer wrapper is `h-screen overflow-hidden`. Adding scroll to the page level or removing `min-h-0` from flex children will break the layout on long content.
- **State machine in `useInterview.ts`** — the `IDLE → PLAYING_QUESTION → READY → RECORDING → REVIEWING → SUBMITTING → FEEDBACK → BREAK` sequence is enforced by `guardState()` guards. Do not add shortcut transitions or skip states. New features that change flow (e.g. skipping intro) must go through the existing state entries, not around them.
- **Zero TypeScript errors policy** — `npx tsc --noEmit` must pass cleanly before any commit. No `any` types. All new hooks, API handlers, and components must be fully typed against the interfaces in `types/index.ts`.
